# Тесты прошивки (план + критерии)

Ниже — набор тестов для проверки прошивки на железе (HIL, Hardware-In-the-Loop).

## 1) Управление кнопкой и записью

### TC-REC-001: старт записи при удержании
- Предусловие: устройство загружено, SD доступна.
- Шаги: зажать кнопку на 2 сек.
- Ожидание: загорается LED, создаётся WAV-файл.

### TC-REC-002: стоп записи при отпускании
- Шаги: отпустить кнопку после записи.
- Ожидание: LED гаснет, WAV закрыт корректно (читается любым WAV-плеером).

### TC-REC-003: ограничение 5 минут
- Шаги: удерживать кнопку дольше 5 минут.
- Ожидание: автозавершение записи, в логе отметка о max duration.

## 2) SD и устойчивость

### TC-SD-001: запись нескольких файлов
- Шаги: записать 10 коротких сообщений.
- Ожидание: 10 отдельных WAV в `/audio`, уникальные имена.

### TC-SD-002: перезагрузка после записи
- Шаги: записать файл, перезагрузить устройство.
- Ожидание: ранее записанный файл не повреждён.

## 3) Синхронизация и сеть

### TC-NET-001: отправка при доступной сети
- Шаги: подключить Wi‑Fi и сервер.
- Ожидание: файлы отправляются, появляются `.sent` markers.

### TC-NET-002: ретрай после недоступности сети
- Шаги: отключить Wi‑Fi, сделать записи, включить Wi‑Fi.
- Ожидание: отложенные файлы отправляются после восстановления.

### TC-NET-003: не дублировать уже отправленные
- Шаги: перезапуск после успешной отправки.
- Ожидание: файлы с `.sent` повторно не отправляются.

## 4) Сквозной сценарий

### TC-E2E-001: голос -> сервер -> Obsidian
- Шаги: записать тестовую фразу, дождаться отправки и ASR.
- Ожидание: создана заметка в Obsidian с корректным текстом/временем.

## Рекомендованная автоматизация
- Серверный endpoint поднять в test-режиме с логированием входящих upload.
- Подключить USB-UART и парсить serial-лог для авто-верификации событий:
  - `[REC] started`
  - `[REC] stopped`
  - `[UP] ... -> HTTP 2xx`
